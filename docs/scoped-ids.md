# Scoped Widget IDs

Named containers automatically scope the IDs of their children. This
gives every widget a unique, hierarchical identity that reflects where
it lives in the tree, without requiring the developer to manually
construct long prefixed strings.

## How it works

When the tree is normalized, any node with an explicit (non-auto) ID
pushes that ID onto a scope chain. All descendant nodes have their
IDs prefixed with the scope path, joined by `/`.

```
sidebar (Container)        ->  id: "sidebar"
  form (Container)         ->  id: "sidebar/form"
    email (TextInput)      ->  id: "sidebar/form/email"
    save (Button)          ->  id: "sidebar/form/save"
```

### What creates scope

- **Named containers** -- any node with an explicit ID (not starting
  with `auto:`) pushes its ID onto the scope chain.

### What does NOT create scope

- **Auto-ID widgets** -- IDs starting with `auto:` (generated by
  the builder layer when you don't provide an ID) pass through
  without adding scope.
- **Window nodes** -- `type: "window"` nodes never scope, since
  windows are logical boundaries, not containment boundaries.

### Slash validation

User-provided IDs cannot contain `/`. The framework throws an error
if a manually slashed ID is detected. The `/` separator is reserved
for the scoping mechanism.

## Events

When the renderer emits an event for a widget, the wire ID is the full
scoped path (e.g. `"sidebar/form/save"`). The protocol decoder splits
it into a local `id` and a `scope` array:

```typescript
// event.kind === "widget"
// event.id === "save"
// event.scope === ["form", "sidebar"]
```

The `scope` array is in **reverse order** -- nearest parent first.
This optimises the common case of checking the immediate parent:

```typescript
// Match on local ID only (ignores scope)
if (isClick(event, 'save')) { ... }

// Check immediate parent
if (isClick(event, 'save') && event.scope[0] === 'form') { ... }

// Bind the parent for dynamic lists
if (isToggle(event, 'done') && event.scope[0]) {
  const itemId = event.scope[0]
  return toggleItem(state, itemId)
}
```

### Reconstructing the full path

Use `target()` to get the full forward-order path:

```typescript

// event.id === "save", event.scope === ["form", "sidebar"]

// => "sidebar/form/save"
```

## Which events carry scope

- `WidgetEvent` (click, input, toggle, etc.)
- `CanvasEvent`
- `MouseAreaEvent`
- `PaneEvent`
- `SensorEvent`

Subscription events (Key, Mouse, Touch, IME, Modifiers) are global
and do not carry scope.

## Canvas element scoped IDs

Canvas elements participate in the same scoping mechanism as regular
widgets. When a canvas element emits an event, the renderer uses
`canvas_id/element_id` as the wire ID. The SDK splits this into
`id: elementId, scope: [canvasId, ...]`, making canvas elements
look like regular widgets inside a container from the event
perspective.

```
Canvas id="drawing"                ->  id: "drawing"
  rect id="bg"                     ->  id: "drawing/bg"
  interactive id="handle"          ->  id: "drawing/handle"
```

Events from interactive canvas elements follow the same pattern:

```typescript
// Canvas element click arrives as:
// { type: 'canvas_element_click', id: 'handle', scope: ['drawing'], ... }

// Check scope in update:
if (isCanvasElementClick(event) && event.id === 'handle' && event.scope[0] === 'drawing') {
  // handle in drawing canvas
}
```

This means canvas elements, canvas widgets, and regular widgets all
use the same scoping and event dispatch model.

## Dynamic lists

When rendering a list of items, wrap each item in a named container
using the item's ID. The container creates a scope for the item's
children, giving each instance unique IDs without manual prefixing.

```tsx
<Column id="todoList">
  {state.items.map(item => (
    <Container id={item.id}>
      <Row spacing={8}>
        <Checkbox id="done" value={item.completed} onToggle={toggleItem} />
        <Button id="delete" onClick={deleteItem}>X</Button>
      </Row>
    </Container>
  ))}
</Column>
```

This produces IDs like `"todoList/item_1/done"` and
`"todoList/item_2/delete"`. In the handler, read the item ID from
the scope:

<!-- test: scoped_ids_dynamic_list_delete -- keep this code block in sync with the test -->
```typescript
const toggleItem = (s: Model, e: WidgetEvent): Model => {
  const itemId = e.scope[0]  // "item_1", "item_2", etc.
  return {
    ...s,
    items: s.items.map(i =>
      i.id === itemId ? { ...i, completed: !i.completed } : i
    ),
  }
}

const deleteItem = (s: Model, e: WidgetEvent): Model => ({
  ...s,
  items: s.items.filter(i => i.id !== e.scope[0]),
})
```

## Scope matching patterns

<!-- test: scoped_ids_depth_agnostic, scoped_ids_exact_depth, scoped_ids_exact_depth_mismatch, scoped_ids_no_scope, scoped_ids_no_scope_mismatch -- keep this code block in sync with the test -->
```typescript
// Depth-agnostic: works whether nested or at root
if (event.id === 'query' && event.scope[0] === 'search') { ... }

// Exact depth: only matches if "search" is the only scope ancestor
if (event.id === 'query' && event.scope.length === 1 && event.scope[0] === 'search') { ... }

// No scope: only matches unscoped widgets
if (event.id === 'save' && event.scope.length === 0) { ... }
```

## Accessibility cross-references

The a11y props `labelledBy`, `describedBy`, and `errorMessage`
reference other widgets by ID. During normalization, these references
are automatically resolved relative to the current scope:

```tsx
<Container id="form">
  <Text id="nameLabel">Name:</Text>
  <TextInput id="name" value={state.name}
    a11y={{ labelledBy: 'nameLabel' }} />
</Container>
```

On the wire, `labelled_by` becomes `"form/nameLabel"`, matching
the scoped ID of the label widget. References that already contain
`/` (full paths) pass through unchanged.
